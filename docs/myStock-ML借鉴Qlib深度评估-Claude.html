myStock ML 如何借鉴 / 使用 Qlib

基于逐文件读 mystock/ml/（15 文件 / 2561 行）+ 三份 ML 文档 + qlib 源码的深度调研 · 2026-07-18

这份报告有一个明确的立场，而不是"qlib 很强，接入它吧"。我读完 mystock/ml/ 全部代码和 ML_PLAN / ML_OVERVIEW / ML_ALGORITHM_PROPOSAL 三份文档后，判断是： myStock ML 与 qlib 存在结构性错位，整体接入是错的；但 qlib 有 3–4 个"点"值得精确借鉴， 其中 2 个值得直接把代码抄过来。 报告的价值在于把这些"点"逐个拆开，给出 借思想 / 抄代码 / 拒绝 三档裁决，并说清每一档背后的理由。

0 · 裁决（一句话）

Verdict 不要接入 qlib 框架。 myStock ML 已经是一套自洽、防泄漏做得比很多人扎实的小系统； qlib 是为"全市场截面选股"设计的重型框架，与 myStock"单标的、独立账户、区间预测"的问题形态不匹配。

但要精确借鉴 qlib 的三样东西：① 表达式化的特征库思路（把 features.py 升级成可组合、可缓存的因子集）； ② 直接抄 Purged/Embargo + 滚动 的评估纪律（这是当前最痛的缺口）； ③ 抄它的"信号 IC 分析 vs 组合绩效"两级评估拆分。 其余（模型园、TopkDropout、嵌套执行器、RL 订单执行）对本项目无用或有害。

这个结论与 myStock 团队自己文档里的保守姿态一致但更进一步：团队已经在 ML_ALGORITHM_PROPOSAL §8 明确否掉了 TFT/Transformer、跨币种归一、跨标的现金池、在线 RL——这些恰好也是"整体 qlib 化"会带来的东西。 本报告的增量是：把"该拒绝什么"讲清楚之外，把"该拿什么"精确到文件级。

1 · 核心论点：错位与真瓶颈

整份评估建立在两个判断上，后面所有裁决都从这里推出来：

判断一：问题形态错位

维度qlib 假设的问题myStock ML 的问题
横向宽度全市场数千标的截面排序选股6 支固定标的，各自独立
预测目标单值 alpha 分数（谁涨得多）次日 [L_hat, H_hat] 区间（波动到哪）
决策形态Topk 组合权重 + 再平衡单标的限价挂价档 + 数量档
资金共享现金池、组合级风险单标的独立账户、本币闭环
样本量数千标的 × 十余年（百万级样本）每股 ~1255 日线（合计几千行）
为什么这决定了一切 qlib 的三大王牌——截面归一化（CSRankNorm）、TopkDropout 组合策略、嵌套多频执行器—— 全部以"多标的截面"为前提。myStock 是 6 条各自独立的时间序列，截面维度退化成 1， 这些王牌要么无从施展、要么退化成空操作。这不是"qlib 不够好"，是用错工具。

判断二：真瓶颈不是模型，是"可信度"

读回测结果（第 3 节）后，一个反直觉但清晰的事实浮现：myStock ML 现在缺的不是更强的模型， 而是"我怎么知道这个结论不是运气"的能力。 6 个标的、单一 regime（2024–2026 强相关）、 walk-forward 只有 4 折——任何"bandit 赢了 1 个标的"的结论都在噪声量级内。 qlib 对本项目最大的价值，恰恰不在它的模型园，而在它把评估纪律工程化的那部分。

2 · ml 代码逐文件审计

先说清楚现状，评估才有据。mystock/ml/ 的实现质量高于预期——尤其防泄漏这一关做得比多数量化教程扎实：

文件职责我的评价
features.py16 个技术特征 + 次日 high/low 标签用 adj_close 算收益、标签用 shift(-1)、最后一行留 NaN 作推理——口径干净。但特征是硬编码函数，不可组合

一句话总评 这套代码的"防泄漏 + 严基线 + 能停则停"的工程纪律，本身就是 qlib 设计哲学的一次独立实现。 换句话说，myStock ML 已经"内生"了 qlib 里最值钱的那部分理念——所以它需要 qlib 的地方，比想象中少。

3 · 回测证据与它在说什么

这是 ML_PLAN.md §10 (P3) 如实记录的关键 go/no-go 证据（494 测试日，单标的独立账户，初始 2 万）：

标的rule (S0)bandit (S2)human 回放buy&hold行情
NVDA25040288582420633638单边强涨
TSLA28244186062542038750单边强涨
PDD19767189742010911840震荡下行

把这张表读透，能得到三个层层递进的结论，它们直接决定"该向 qlib 要什么"：

  1. 单边牛市里 buy&hold 几乎不可战胜（NVDA/TSLA）。这不是模型弱——是任何择时在强趋势里都在做负和的事。qlib 换再强的模型也改不了这条物理规律。
  2. 震荡/下行里择时有真实价值（PDD：所有交易策略都赢了 buy&hold 的 11840）。这说明系统的信号本身是有效的，只是被 2 只趋势股的行情掩盖。
  3. bandit 尚未稳定打过 S0 规则（只赢 NVDA）。这才是决策层的真问题——而它大概率是样本/方差问题，不是算法容量问题。
关键推论 结论 ①②③ 全都指向同一件事：在 6 标的 × 单一 regime × 4 折的评估强度下，我们分不清"bandit 输了"是算法差还是运气差。 这就是为什么第 4 节说真瓶颈是"可信度"。先把评估做扎实，再谈换模型，否则换了也不知道有没有用。

4 · 真正的瓶颈在哪

把当前系统的"缺口"按"对结果的影响 × 修复成本"排序，答案很不像直觉——排在最前的都不是模型：

缺口影响qlib 能帮上忙吗
评估方差不可控：4 折、无 embargo、种子少致命✅ 直接抄它的 Purged/Embargo 思路 + 滚动多折
信号质量与策略绩效未分离：只看净值，不看 IC高✅ 抄 SignalRecord / SigAnaRecord 的两级评估
特征不可组合、难扩展：16 个硬编码函数中◐ 借它表达式引擎的思路（不必抄库）
模型容量（LGBM 分位数回归）低✗ qlib 的深度模型在此数据量下大概率更差
决策算法（LinUCB → ？）低✗ qlib 无对应组件；且不是当前瓶颈
撮合保真（1h → 15m）中✗ 与 qlib 无关（qlib 的 Exchange 是日频/组合级）
这张表就是全文的地图 影响"致命/高"的两行，qlib 都能帮上——但帮的方式是抄它验证/评估层的代码与纪律，不是用它的模型。 影响"低"的三行，恰好是外行最容易想"上 qlib 换个 SOTA 模型"的地方——而它们对本项目基本无用甚至有害。 这个反差，是本报告最想传达的东西。

5 · 概念对照：ml ↔ qlib

把 myStock ML 的每个模块，对到 qlib 的等价物，能一眼看出哪里"已经有了"、哪里"qlib 更成熟"、哪里"根本对不上"：

myStock/ml Qlib 等价物 关系 ────────────────────────────────────────────────────────────────────────── features.py (硬编码 16 特征) → data/ops.py 表达式引擎 qlib 更强（可组合/缓存）→ 借思路 predictor.py (分位数回归) → contrib/model/gbdt.py (LGBM) 等价，ml 已直接用 LGBM → 无需借 calibrator.py (CQR) → (qlib 无原生 CQR) ml 反而更前沿 → 无可借 regime.py (HMM 无泄漏前向) → contrib/meta/ (DDG-DA 漂移) 思路同源，实现各异 → 参考不照搬 policy.py (LinUCB/规则) → contrib/strategy/signal_*.py 形态不同（区间挂价 vs 组合权重）→ 对不上 simulator.py (1h 限价撮合) → backtest/executor.py (嵌套) qlib 是组合/日频 → 对不上 backtest.py (三基线净值) → workflow/record_temp.py (记录链) qlib 的两级评估更成熟 → 抄纪律 offline_rl.py (CQL) → rl/ (订单执行 RL) 目标不同（择价 vs 拆大单）→ 对不上 (无) → Purged/Embargo CV ml 缺，qlib/学界成熟 → 抄 (无) → SignalRecord IC 分析 ml 缺 → 抄

这张图揭示一个有意思的结构：myStock ML 在"预测层"甚至比 qlib 更前沿（CQR、区间预测）， 在"决策层"与 qlib 形态完全不同（对不上），真正落后的只有"评估层"。 所以借鉴的重心应压在评估层，而不是散射到各处。

对 qlib 的每个主要能力，给出明确裁决。图例： 借思想（读其设计，自己实现）· 抄代码（直接移植/vendored 其实现或算法）· 观望（扩样本后再议）· 拒绝（对本项目无用/有害）。

qlib 能力裁决理由
表达式因子引擎（data/ops.py）借思想把 features.py 重构成可组合/可缓存的"因子表"，利于批量试因子；但不必引入 qlib 的 bin 数据服务，本地 SQLite 上自建轻量版即可

信号 IC 分析（SigAnaRecord）抄代码把"只看净值"升级为"IC/RankIC 看信号 + 净值看策略"两级评估，能定位 bandit 到底是信号问题还是执行问题
市场漂移/元学习（contrib/meta DDG-DA）借思想regime.py 已是同源思路；可参考 DDG-DA 把"预测分布会怎么漂"也建模，但当前 6 标的样本量不足以支撑，列为储备
模型园（LSTM/GATs/HIST/TFT…）拒绝团队文档已否 TFT/Transformer；几千行样本上深度模型输给 LGBM 是常态。换模型不是当前瓶颈

RL 订单执行（qlib/rl）拒绝目标不同：qlib RL 拆大单择时，ml 是择价；且团队已把 RL 定位研究性、不强推
RD-Agent（LLM 自动因子）观望需先有可组合的表达式因子层（见第 1 条）+ 足够回测吞吐；是"因子层做好后"的远期放大器，非现在
qlib 数据服务 / bin 格式拒绝会与 myStock"独立 SQLite + web 只读"边界冲突；引入成本远超收益

7 · 该借鉴的四件事（借思想）

① 把 features.py 升级成"可组合因子表"

现在的特征是 16 个写死在函数里的列。qlib 的洞见是因子 = 声明式表达式，好处是可批量生成、可版本化、可缓存。 不必引入 qlib 的库，只需在本地把因子登记成一张名字 → 计算函数的注册表： 

# 借 qlib 思想的轻量因子注册表（自实现，不依赖 qlib）
FACTORS = {
    "ma5_dev":   lambda d: d.adj/d.adj.rolling(5).mean() - 1,
    "vol_20d":   lambda d: d.ret1.rolling(20).std(),
    "corr_pv20": lambda d: d.ret1.rolling(20).corr(np.log(d.volume+1)),  # 新因子零成本加入
    ...
}
# 试新因子 = 往表里加一行 + 重跑 walk-forward，不改建模代码

收益：把"加因子"从改代码降为改配置，为后续 A/B 试因子、乃至远期 RD-Agent 式自动挖因子铺路。成本：半天重构。

② 两级评估：IC 看信号，净值看策略

qlib 的 SignalRecord → SigAnaRecord → PortAnaRecord 把"信号好不好"和"策略赚不赚"拆成两级。 myStock 现在只有第二级（净值）。补上第一级（对区间中点或方向算 IC / RankIC）， 就能回答第 3 节那个悬案：bandit 输，是信号没预测力，还是执行没把信号用好？

③ 参考 DDG-DA 的漂移建模（储备，不落地）

regime.py 已经在做"识别当前状态"。qlib 的 DDG-DA 更进一步——建模"分布将要怎么漂"。 当前 6 标的样本撑不起，但作为方法储备值得知道：扩标的后，这是 regime 之上的下一步。

④ 借它"配置即实验"的可复现纪律

qlib 一份 YAML = 一次可复现实验。myStock 的 BTConfig 已经很接近（dataclass 全参数化）。 值得补的是把每次回测的完整配置 + 数据窗口 + 种子 + git commit 落进报告头部， 让 data/ml/reports/<date>/ 的每份快照都能被精确重跑。这是 qlib Recorder 的平民版。

8 · 该"抄代码"的两件事（最高优先级）

这是本报告最可操作的部分 下面两件事修复的是第 4 节里"影响=致命/高"的两行。它们不引入 qlib 依赖——算法都只有几十行， 自实现即可，但"抄 qlib/学界成熟做法"能避免踩坑。建议在换任何模型之前先做这两件。

① Purged K-Fold + Embargo（防泄漏评估的地基）

当前 walk_forward_eval 是标准滚动切分，但训练段末尾和测试段开头之间没有"隔离带"。 由于特征用了 20 日滚动窗口、标签用次日收益，训练/测试边界处存在信息重叠——评估会系统性偏乐观。 López de Prado 的 Purged CV + Embargo 正是修这个（qlib 生态也采用）：

收益：让"bandit 赢没赢"的结论第一次可信。这直接把第 3 节的悬案变成可判定的问题。成本：1–2 天。 这也正是团队 ML_ALGORITHM_PROPOSAL 建议 9 已排进"第 1 周"的事——本报告把它标为头号优先级。

② 信号 IC / RankIC 分析器

抄 SigAnaRecord 的口径：对每日预测（区间中点收益、或"次日会不会突破区间中点"的方向） 与真实次日收益算截面/时序 IC、RankIC、ICIR。虽然 myStock 是单标的（截面退化）， 但可在时间维算滚动 IC，回答"预测信号随时间是否稳定有效"。约 30 行纯函数，天然可单测， 与现有 calibrator.py 的风格完全一致。

9 · 该拒绝 qlib 的地方（同样重要）

"不做什么"和"做什么"一样是决策。以下 qlib 能力，对本项目无用或有害，明确不采纳——这与团队 ML_ALGORITHM_PROPOSAL §8 的"不推荐清单"互相印证：

拒绝为什么对 myStock 是错的
整体接入 qlib 框架问题形态错位（第 1 节）；且违反"独立 ML 库 + web 只读"边界，工程成本远超收益
qlib 深度模型（TFT/LSTM/GATs）几千行样本上噪声 > 信号，SOTA 深度模型输给 LGBM 是常态；换模型不是瓶颈
截面归一化（CSRankNorm）单标的下截面维度=1，归一化退化成空操作
TopkDropout / 组合优化器无"选哪些股"的问题（6 支全持），无组合权重可优化
把 qlib 塞进 web 请求路径qlib 重计算 + 缓存，绝不能进 Flask 只读路由（违反 CLAUDE.md 边界）
为用 qlib 而跨币种归一成 USD违反"各股本币闭环、不换汇"锁定；港美股必须分开
一条容易犯的错 最诱人的错误是"既然回测打不过基线，那上个 qlib SOTA 模型试试"。第 3、4 节已论证：打不过 buy&hold 是趋势物理规律， 打不过 S0 大概率是评估方差——两者都不是模型容量问题。先上 qlib 模型 = 在没修好尺子之前先换砝码， 结果依然读不出真信号。

10 · 落地顺序与验收

把上面所有裁决压成一条可执行、带 go/no-go 的顺序。刻意与团队现有路线图对齐（不另起炉灶）：

第 0 步（先修尺子，最高优先，~2 天） ├─ 抄 Purged+Embargo CV → 门槛：5 种子 std/mean ≤ 0.3 [对应团队建议 9] └─ 抄 IC/RankIC 信号分析器 → 门槛：能区分"信号无效" vs "执行无效" 第 1 步（重估现有结论，~1 天） └─ 用新尺子重跑 P3 回测 → 判定 bandit 到底有没有真的输给 S0 （很可能结论会变：噪声被压下来后） 第 2 步（借因子思想，~半天） └─ features.py → 因子注册表 → 门槛：加一个新因子 = 改一行配置 第 3 步（团队既定路线，qlib 不介入） └─ 继续 建议1/2/3（reward/CQR/regime）已在做，按团队门槛推进 储备（扩标的后再议） └─ DDG-DA 漂移建模 / RD-Agent 自动因子 / 15m 撮合
与团队路线图的关系 本报告不新增算法负担。它做的是给团队既有的"建议 9（多种子+embargo）"提级为头号优先， 并明确它应先于建议 1/2/3 的效果评估——因为没有可信的尺子，建议 1/2/3 的"提升"也读不准。 其余全部沿用团队既定门槛：独立 ML 库 永留 S0+三基线 打不过就停 红涨绿跌 web 只读。

11 · 收尾：一个反直觉结论

大多数"要不要用 qlib"的评估，会花大篇幅比较模型园、羡慕 SOTA。这份报告的结论正相反：

The Takeaway myStock ML 不需要 qlib 的"力量"，只需要 qlib 的"纪律"。

它的预测层（CQR 区间预测）已经比 qlib 更前沿；决策层与 qlib 形态不同、无从借；模型层换 qlib 也不会更好。 唯一真实的缺口在评估的可信度——而这恰好是几十行代码、零框架依赖就能补上的。 把 qlib 当"评估纪律的参考实现"，而不是"模型工厂"，才是对这个项目正确的用法。

这也解释了为什么团队自己的文档已经走到了很保守的位置——他们在正确的方向上。本报告只是把"该借什么" 从"读读 qlib 找灵感"收敛到两个文件级的、可立刻动手的改动，并给出为什么其余的都该放下。
