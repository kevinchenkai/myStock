myStock · 项目接手指南

个人港股 / 美股 持仓、交易、行情的本地化数据系统 —— 一份从零到接手的完整导读

Python 3.10+ Flask SQLite futu-api yfinance 单机 / 单用户 / 无鉴权

1 · 项目是什么

一句话：把分散在 富途（持仓 / 历史订单 / 历史成交）和 yfinance（每日行情 / 公司信息）的数据，统一抓进本地 SQLite，再用本地 Flask 页面查询、下钻到单只股票、计算交易盈亏与复盘。

定位与边界

  • 单用户、单机、无登录鉴权。Web 服务默认只监听 127.0.0.1:8888。
  • 市场范围仅 HK（港股）与 US（美股），无 A 股、无其它市场。
  • Web 层只读 SQLite，绝不直接调富途 / yfinance —— 保证页面快、可离线。
  • 新增抓取功能 → 放进 pipeline；新增展示功能 → 只读库。这条边界不能破。

它解决了什么

  • 富途只给"当前"持仓，历史快照不可回补 —— 本项目随 update.sh 逐日自然积累。
  • 富途界面看不到"已实现盈亏 / 回合复盘 / 年度现金流 / 资产趋势"等聚合视图 —— 本项目自己算。
  • 跨数据源（富途 + yfinance）的代码、币种、时间口径不统一 —— 本项目用 code_map.py 与 currency 字段统一。

阅读建议

先 §1–§4 建立全局观 → §5–§11 逐模块读源码 → §12–§15 建立数据与口径直觉 → §16+ 了解 ML 扩展与运维。CLAUDE.md、README.md、docs/DATA.md 是权威文档，本指南是它们的导读与索引。

2 · 架构与数据流

富途 OpenD（本地网关 127.0.0.1:11111）──┐ futu-api │ yfinance（行情 / 通用信息）──────────────┴──► collectors ──► SQLite (data/mystock.db) ──► Web (Flask :8888) 采集层 本地库 只读展示层

三层职责

层位置职责能否离线
采集层mystock/collectors/从富途 OpenD 与 yfinance 拉数据，清洗后写库否（需网络 / OpenD 登录）
存储层data/mystock.db (SQLite)8 张表，全部 UPSERT 幂等是
展示层mystock/web/Flask 路由 + 原生 JS 前端，只读库是

典型数据流（以"持仓"为例）

  1. scripts/update.sh → mystock.pipelines.update_load
  2. 调用 collectors.futu_client.FutuClient 连本地 OpenD，position_list_query
  3. futu_client.position_rows() 把 DataFrame 规整成入库 dict
  4. db.replace_position_snapshot()：先删除当天，再 UPSERT（保证已清仓标的不残留）
  5. 用户浏览器 → GET /api/positions → Flask 查最新快照 → JSON 返回 → app.js 渲染表格
边界铁律 展示层 只读 SQLite。任何"在页面上点一下触发抓取"的需求都不应实现 —— 抓取永远走 pipeline 脚本。这保证页面快、可离线、可重复。

3 · 5 分钟跑起来

前置条件

  • 本机已装 conda（Miniconda / Anaconda 均可）。
  • 富途 OpenD 已启动并登录（默认 127.0.0.1:11111）。下载：openapi.futunn.com。否则富途数据抓取会失败。
  • 交易密码：推荐用环境变量 export MYSTOCK_FUTU_TRADE_PWD='你的交易密码'，不要写进文件、不要进 git。

三步脚本（顺序：init → update → server）

# 1) 首次全量：建环境 + 建库 + 全量抓取（幂等，可重复）
bash scripts/init.sh

# 2) 之后不定期增量更新（需 OpenD 已登录）
bash scripts/update.sh

# 3) 启动 Web（仅读库，不抓取）
bash scripts/server.sh
# 浏览器打开 http://localhost:8888
注意 所有脚本会自动 conda activate mk。改了后端路由 / pnl 逻辑后必须重启 server.sh（运行中的 Flask 进程不会热加载）；改前端 JS/CSS 后刷新浏览器即可，无需重启。

测试

conda activate mk
python -m pytest tests/ -q

测试必须在 mk 环境跑（base anaconda python 无 yfinance 会报 YFError）。现有测试覆盖 code_map / db / pnl / yf_client 及 ML 子模块。

6 · code_map.py —— 代码映射

纯函数模块，富途代码 ↔ yfinance 代码互转。是跨数据源关联的枢纽，有完整单测。

市场富途yfinance规则
港股HK.007000700.HK去 HK. → 数字规整为 4 位 → 加 .HK
美股US.AAPLAAPL去 US.，直接用 ticker

核心函数

  • futu_to_yf(futu_code)：HK.00700 → 0700.HK，US.AAPL → AAPL
  • yf_to_futu(yf_symbol)：反向；港股富途用 5 位（0700.HK → HK.00700）
  • normalize_hk_number(num)：港股数字去前导 0 后补足 4 位（00700 → 0700，09988 → 9988）
  • futu_market_of(futu_code)：从代码取 HK / US 前缀
跨表 JOIN 一律用富途代码 daily_quotes 与 stock_profiles 同时有 yf_symbol 和 futu_code 两列，但跨表关联统一用 futu_code，避免位数/后缀差异。

7 · db.py —— 数据库封装

所有写库走 UPSERT（INSERT ... ON CONFLICT(...) DO UPDATE），天然幂等，可重复执行。

核心 API

函数用途
get_connection()返回 SQLite 连接，row_factory=Row（按列名访问），自动建父目录
init_db()执行 schema.sql，IF NOT EXISTS 可重复
_upsert(conn, table, rows, conflict_keys)通用 UPSERT，按主键冲突时更新非主键列
upsert_positions / orders / deals / quotes / profiles / fx_rates各表的语义化封装
replace_position_snapshot(conn, date, rows)持仓专用：先 DELETE 当天再插入，保证已清仓标的不残留
write_sync_log / last_sync_point同步日志；增量更新据此取上次 range_end
all_traded_codes(conn)从 positions/orders/deals 去重得到全部富途代码

跳过名单（quote_skiplist）机制

对连续抓不到数据的标的计数，达阈值 SKIP_THRESHOLD = 5 后加入名单，后续直接跳过，避免无效请求与退市噪音。

  • record_quote_empty(conn, code, yf_symbol)：空数据计数 +1
  • get_quote_skiplist(conn)：返回 empty_count >= 5 的代码集合
  • clear_quote_skip(conn, code)：重新抓到数据则移出名单
  • reset_quote_skiplist(conn, codes=None)：手动重置（None=全清，或指定代码）
  • purge_code(conn, code)：不可逆，从所有表彻底删除某代码（会改历史盈亏）
不自愈 进入名单的代码不再请求 → 无从触发 clear。一次限频/网络抖动可能批量抓空误伤真实持仓。误伤后手动止血：
python -m mystock.pipelines.maintenance reset-skiplist [代码...]

8 · collectors —— 采集层

8.1 futu_client.py —— 富途

mystock/collectors/futu_client.py
通过本地 OpenD 网关通信，不直连富途服务器。历史成交仅支持 TrdEnv.REAL。
  • 限频控制：富途历史订单/成交接口限频"每 30 秒最多 10 次"。RATE_LIMIT_INTERVAL=3.2s 窗口间隔，命中限频关键字自动退避 RATE_LIMIT_BACKOFF=30s 重试（最多 5 次）。
  • 80 天窗口分段：WINDOW_DAYS=80，_date_windows(start, end) 把长区间切成 ≤80 天窗口逐段查询后 pd.concat 合并（单次默认仅 90 天）。
  • FutuClient(market)：上下文管理器，__enter__ 连接 + _maybe_unlock，__exit__ 关闭。
  • 三个查询：query_positions() / query_orders(start, end) / query_deals(start, end)。
  • 三个规整函数：position_rows() / order_rows() / deal_rows() 把 DataFrame 转入库 dict；_g(row, *names) 按候选列名取首个非空值（兼容富途字段名变化）。
  • raw_json：原始记录整行 JSON 化后入库，schema 之外的信息可后续解析。

8.2 yf_client.py —— yfinance

mystock/collectors/yf_client.py
日线 + 外汇 + 公司信息。带重试与退避，yfinance 自身日志级别抬高至 CRITICAL（退市噪音由 skiplist 接管）。
  • _end_inclusive(end)：yfinance 的 history(end=...) 是排他的（只返回 < end 的 bar），传 end=今天 会漏当天。本函数把 end + 1 天修正，确保抓到"当天"这根 bar。日线与汇率统一经此修正。
  • fetch_daily(futu_code, start, end)：单标的日线，auto_adjust=False 同时保留 close 与 adj_close，含 dividends / stock_splits。
  • fetch_fx(yf_symbol="CNY=X", pair="USDCNY")：外汇日线，close = 1 美元对应的人民币。外汇对仅有 OHLC，无成交量/分红。
  • _profile_from_info(info) + fetch_profile(futu_code)：从 Ticker.info 提取公司/估值字段；全为空视为无有效资料（退市），返回 None 不入库。
单标的失败不中断整体 fetch_daily 抛异常会被 pipeline 捕获、记 sync_log 后继续下一个标的。这是"失败可恢复"原则的体现。

9 · pipelines —— 流水线

9.1 init_load.py（全量，init.sh 调用）

流程：init_db() → collect_positions → collect_orders(start, end) → collect_deals → collect_quotes(start, end_date) → collect_fx → collect_profiles（放最后，依赖 quotes 已更新的跳过名单）。每步写 sync_log。

collect_quotes 会先 get_quote_skiplist 过滤，对抓到空的标的 record_quote_empty，抓到数据则 clear_quote_skip。结束汇总 X ok / Y empty / Z err / N skipped。

9.2 update_load.py（增量，update.sh 调用）

复用 init_load 的各 collect_*，仅改起点：

  • _incremental_start(conn, source, fmt)：取上次成功同步的 range_end 的日期部分当天 00:00（保证当天数据被重抓覆盖）；无记录则回退 start_date。
  • 持仓：始终覆盖当天快照。
  • 订单/成交：从"上次同步点当天 00:00"重抓（UPSERT 天然幂等）。
  • 行情/汇率：从上次 range_end 当天重抓（PK 覆盖）。
  • profiles：每日全量刷新（UPSERT 覆盖）。

10 · pnl.py —— 盈亏计算（项目核心算法）

两套口径并存

函数口径用途
compute_pnl(deals, cost_fallback)移动平均成本（券商口径）盈亏 Tab 列表（每股一行）
analyze_stock(deals, cost_fallback)FIFO 配对（干净的持有周期）单股复盘浮窗（回合 + 胜率/盈亏比）
yearly_finance(deals, year)年度现金流（卖出额 − 买入额）财务统计板块，按美股/港股分别汇总

移动平均成本法（compute_pnl）

  • 同一标的按 create_time 升序处理。BUY：累加 held_qty 与 held_cost。
  • SELL：先用窗口内持仓（avg = held_cost / held_qty）配平，realized += (price - avg) * matched，按比例扣减基数。
  • 成本兜底：抓取起点（start_date）前的建仓买入不在库里 → 卖出量 > 窗口内买入量时，剩余用 positions.cost_price 兜底。
  • 兜底成本 ≤ 0 视为不可用（富途超卖记账产物，如 US.AAPL = -1233.96），对应数量记入 uncovered_sell_qty，不计入盈亏，对用户透明。
  • 已实现盈亏率：相对"已卖出部分的成本"，避免被未平仓部分稀释。
  • 默认按 realized_pnl 降序输出。

FIFO 回合复盘（analyze_stock）

  • 维护买入 lot 队列 [[qty, price, time], ...]，卖出时从队首按 FIFO 配对，每对记一回合（含 hold_days、pnl、pnl_ratio）。
  • 剩余卖出无对应买入：兜底成本配一回合（标 fallback=True），否则记 uncovered_sell_qty。
  • stats：胜率、盈亏比（profit_factor = 总盈利/总亏损）、平均/最大/最小持有天数、最大盈/亏单。
  • _build_observations：生成客观交易行为复盘提示（good / warn / info），仅事实陈述，不输出投资建议（非持牌投顾）。
口径差异 移动平均成本与 FIFO 配对结果会有小差异 —— 建模前先明确口径。盈亏均为税前毛额（无费用字段，raw_json 也没有费用明细）。

币种

deals 表无 currency 列，pnl.py 用 _MARKET_CCY = {"HK": "HKD", "US": "USD"} 按 market 推断。合计按币种分别汇总，HKD / USD 不混算。

12 · 数据库表结构（schema.sql）

13 · 数据字典速查

仅列字段含义与必须知道的坑。完整字段、取值范围、缺失率见 docs/DATA.md。

positions —— 持仓每日快照

  • snapshot_date / market / code（主键三件套）、name / qty / can_sell_qty / nominal_price / market_val / pl_val / currency / updated_at
  • cost_price：可能 ≤ 0（富途超卖/融券会计产物，观测 7 只美股为负，如 US.AAPL=-1233.96）。建模时 ≤0 视为缺失。
  • pl_ratio：百分比数值（6364.24 表示 +6364.24%），非小数；异常大值通常对应近 0 成本。
  • 历史快照不可回补（富途只给"当前"），随 update.sh 逐日积累。

orders —— 历史委托订单

  • order_id（主键）、trd_side(BUY/SELL)、order_type、order_status、price / qty / dealt_qty / dealt_avg_price、create_time / updated_time、raw_json
  • order_status 枚举：FILLED_ALL / CANCELLED_ALL / CANCELLED_PART / FAILED
  • 派生：是否成交 = order_status in (FILLED_ALL, CANCELLED_PART) 且 dealt_qty>0；撤单率、下单→撤单时长等行为特征可从此表算。

deals —— 历史成交回报（盈亏最权威）

  • deal_id（主键，库内为字符串）、order_id、trd_side、price / qty / create_time、raw_json
  • 无 currency 列 —— 用 market 推断或 JOIN orders/positions。
  • 数据从 2025-01-01 起，更早买入缺失 → 卖出 > 买入的标的需 positions.cost_price 兜底。
  • 无手续费/税费字段 → 盈亏均为税前毛额。
  • create_time 毫秒级、交易所当地时区、无时区标记。跨市场排序前需统一时区。

daily_quotes —— 日线行情

  • yf_symbol / futu_code / date、open/high/low/close（原始价）、adj_close（复权）、volume / dividends / stock_splits
  • 算收益率/技术指标用 adj_close（避免分红拆股日的假跳空）；展示原始价用 close。
  • 各标的交易日数不等长，按 date 取交集或显式处理缺口。

stock_profiles —— 公司/估值

  • market_cap_mm / trailing_eps / target_mean_price 是标的本币（HK→HKD、US→USD），不是美元。跨市场比较前必须换汇。
  • dividend_yield 是百分比数值（8.32 表示 8.32%）。
  • 无历史：每天覆盖，只有最新值。需要估值时序必须自建快照归档。

fx_rates —— 外汇日线

  • pair(USDCNY) / date / open/high/low/close，close = 1 美元对应的人民币。
  • 当天未收盘时 close 可能为空，前端/分析须过滤（注意 JS 中 Number(null)/Number("")===0，不能只用 isFinite）。
  • CNY=X 即 USDCNY；勿与 CNH=X（离岸）混淆。

14 · 关键口径与约定

14.1 涨跌配色（全站统一，中国习惯）

红 = 涨，绿 = 跌，0 为中性灰。切勿用国际惯例反过来。前端统一走 plClass() 与 .up/.down/.flat CSS class。

14.2 货币

yfinance 的 market_cap / EPS / 目标价为标的本币（HK→HKD、US→USD），非 USD。展示时按 currency 字段标注，勿硬编码美元。跨币种（USD/HKD）不相加，按市场各画一条线 / 各一张卡。

14.3 幂等与覆盖

  • 所有写库用 UPSERT；当天可变数据（持仓快照、行情）以覆盖为准。
  • 持仓用 replace_position_snapshot：先 DELETE 当天再 INSERT，保证已清仓标的不残留。
  • init.sh / update.sh 可重复执行，不会产生重复数据。

14.4 资产趋势 / 组合概览口径

均从已入库 positions 快照聚合，零新抓取（Web 只读边界）。跨币种不相加，按市场各画一条线 / 各一张卡；快照不可从富途回补，历史空缺只能随 update.sh 自然积累。

14.5 复盘范围

analyze_stock 仅做客观交易行为复盘，不输出个性化投资建议（你不是持牌投顾）。

15 · 已知坑点（踩坑预防清单）

1. cost_price ≤ 0 富途超卖/融券标的成本可能为负（会计产物，非真实成本）。cost_price <= 0 一律视为缺失，不可直接当成本用。
2. yfinance end 排他 Ticker().history(start, end) 的 end 排他（只返回 < end 的 bar），传 end=今天 会漏当天。统一经 _end_inclusive（end+1 天）修正。
3. 跳过名单不自愈 进入 skiplist 不再请求 → 无从触发 clear。一次限频批量抓空会误伤真实持仓。阈值取 5 留容错；误伤后 maintenance reset-skiplist 手动止血。
4. 路由 <path:code> 吞子路径 用 <code> 不用 <path:code>，否则 /stock/HK.00700/profile 会把 code 解析成 HK.00700/profile。
5. 历史成交仅实盘 history_deal_list_query 仅支持 TrdEnv.REAL，模拟环境不支持。配置 trd_env: REAL。
6. 富途限频 历史订单/成交接口"每 30 秒最多 10 次"。代码按 80 天窗口分段、窗口间 3.2s 间隔、命中限频 30s 退避重试。
7. 持仓快照稀疏 当前库快照日少，时序分析需先积累。历史空缺只能随 update.sh 自然积累，不可回补。
8. 时区无标记 成交/订单时间为交易所当地时区（HK=Asia/Hong_Kong，US=America/New_York）且库内不带时区标记。跨市场按时间排序前先统一时区。
9. purge 不可逆 maintenance purge <代码> 删所有表数据，会改历史盈亏。仅退市清仓且不再关注时用。

16 · ML 子系统概览

独立于 web 生产系统，是后续研究性扩展。完整方案见 docs/ML_PLAN.md，速览见 docs/ML_OVERVIEW.md。

核心思路：拆两层

层干什么方法
预测层（S1）预测次日 [L̂, Ĥ] 区间监督学习 / 分位数回归
决策层（S2/S3）挂什么价、挂多少量规则 → bandit → RL

工程解耦（关键边界）

  • 独立库：data/ml/mystock_ml.db，与 web 生产库 mystock.db 完全解耦。
  • 独立采集脚本：scripts/ml.sh（data/train/publish/all），不并入 update.sh。
  • 独立 conda env：mystock-ml（H20 GPU 机器，与 base/vllm/comfyui 隔离）。
  • 全程不碰 web/生产库；达标稳定后（P6）才把推理结果写入 web 只读库。

数据扩充

  • 5 年日线（1255 行/股）→ 预测层样本 ×3.4
  • 2 年 1h 线（~5000 行/股）→ 撮合模拟器的盘中真实路径，覆盖全部成交窗口
  • 标的：3 美（NVDA/TSLA/PDD）+ 3 港（HK.00700/09988/01810），各股独立账户、本币闭环

进度与结论（诚实记录）

  • P1 撮合校准吻合率 88–90%（真实 orders 回放）→ 撮合可信。
  • P2 预测层成立：次日区间命中 ~50%（按股自适应分位收窄后宽度↓25-30%）。
  • P3.1 bandit 涨势局部有效：NVDA 超 buy&hold；TSLA 改善仍输；PDD 退化。regime 依赖。
  • P4 离线 RL（CQL）无效（负结果）：三支全退化为"全程不动"，conservative_loss > td_loss。符合方案预判 —— 小样本是硬天花板。
  • go/no-go：停在 S1+S2 交付（预测层 + 规则/超额-bandit + 每日报告）。RL 记为已验证的负结果。

关键洞见

  1. 拆两层的设计被验证是对的 —— 预测层稳、可独立交付；避免端到端 RL 黑箱。
  2. 奖励对齐比换更复杂的模型更管用 —— 仅把奖励改成"相对 buy&hold 超额"，NVDA 就从输变赢。
  3. 小样本是硬天花板，RL 学不出有效策略不是调参问题，是数据量问题。
  4. 没有单一策略通吃：涨势 buy&hold 难超越；震荡/下行择时才有价值。下一步 = regime 感知 > 扩样本 > RL。

产物

每日生成自包含 HTML 报告，归档 data/ml/reports/<date>/，红涨绿跌、不碰 web。cron 每工作日早 8 点（北京时间）跑 ml.sh all。

17 · 运维与维护

例行流程

  1. 确保富途 OpenD 已启动并登录。
  2. bash scripts/update.sh 增量更新。
  3. bash scripts/server.sh 启动 / 重启 Web。

常见问题

现象排查
页面无数据 / /api/* 返回 503数据库不存在，先 bash scripts/init.sh
富途数据抓取失败确认 OpenD 已启动登录、端口与 config.yaml 一致；查 sync_log 的 error 记录
历史成交为空确认 trd_env: REAL（成交接口仅支持实盘）
某些股票没行情退市/yfinance 无数据 → 进 quote_skiplist，属正常；个股详情页会提示"行情数据不足"
真实持仓被误跳过限频批量抓空误伤 → maintenance reset-skiplist [代码...]
退市清仓想彻底清除maintenance purge <代码>（不可逆，会改历史盈亏）

sync_log 用途

每次抓取写一条，含 source / range_start / range_end / row_count / status / message / run_at。增量更新据此取上次同步点；排查时按 status='error' 过滤。

18 · 安全与隐私（提交前务必检查）

  • data/、*.db*（真实持仓/交易数据）
  • 交易密码只走环境变量或 config.yaml，绝不硬编码、绝不进 git。
  • 提交前确认 diff 与新文件无密钥 / token / 密码。
  • ML 产物（报告）含真实交易信息，data/ml/reports/ 已被 .gitignore 的 data/ 规则覆盖；公网发布须由用户自己执行（不经 Claude，避免数据外泄闸）。
