## 一、核心思路（一句话 + 五个要点）

> **把"预估次日高低点"（监督、可验证）和"挂价决策"（规则→bandit→RL）拆开，用 1h 撮合 + 三基线把每一步的可信度钉死，能停则停、不为 RL 而 RL。**

### 1. 要复刻的人工动作
你现在：**看几个月趋势 → 预估次日最高/最低点 → 挂限价单（买偏低、卖偏高）+ 决定数量**。
目标函数 = **最大化达成交易净值（卖出额 − 买入额，税前毛额，移动平均成本口径）**。

### 2. 灵魂决策：拆成两层（不做端到端 RL 黑箱）

| 层 | 干什么 | 方法 | 为什么这么分 |
| --- | --- | --- | --- |
| **预测层** | 预测次日 [最低 L̂, 最高 Ĥ] 区间 | 监督学习 / 分位数回归 | 每天一个样本、样本相对足、可单独算误差、可解释 |
| **决策层** | 挂什么价、挂多少量（含不挂） | 规则 → bandit → RL | 这才是序贯/带约束的难点；把不确定性关在这一层 |

拆层的理由：预测层稳、可独立上线；决策层的风险被隔离。**一开始就端到端 RL → 无法 debug + 小样本必过拟合。**

### 3. 务实迭代，能停就停（RL 是研究性增量，不是起点）
```
S0 规则基线（永远保留做对照）
 → S1 预测器（监督，已可独立产出价值）
   → S2 bandit（LinUCB，比 RL 稳、样本效率高）
     → S3 RL（离线 RL / PPO，研究性，可能不达标）
```
**每阶段都有 go/no-go：打不过基线就停在上一阶段交付。**

### 4. 可信度的两个命门
- **撮合模拟器**：无逐笔数据 → 用 **1h K 线近似盘中限价撮合**（能判断盘中先触低还是先触高）。靠**真实 orders 回放校准**——实测吻合率 **88–90%**。
- **三基线门槛**：所有结果对照 ① 买入持有 ② 你的真实历史成交回放 ③ S0 规则。**绝不只报一条精挑曲线。**

### 5. 贯穿始终的纪律
- **防未来函数**：决策只用截至当日收盘可得信息；切分一律按时间 walk-forward，绝不随机打散。
- **敬畏小样本**：真实成交每标的仅 40 多笔 → 优先稳健可解释方法、强正则、多种子看方差。
- **单标的独立**：各自账户、各自本金，不共享现金池（第一版）。
- **工程解耦**：独立库 + 独立采集脚本 + 独立 conda env，**全程不碰 web/生产库**（写 web 是最后的 P6）。

### 各阶段实际"吃"了多少
- **P1 撮合校准**：真实 `orders` 72/74/59 单回放，全部落在 1h 覆盖窗内（无 bar=0）。
- **P2 预测器**：每标的 1255 日线 → 去特征窗(~20)+次日标签 → **≈1234 个可决策样本/标的**，walk-forward 4 折。
- **P3 回测**：`train_frac=0.6` → 训练 ≈740 日、**测试 ≈494 日**，三支独立账户各初始 2 万。

### 三个必须记住的数据约束（影响结论可信度）
1. **成交样本很小**（每标的 40 多笔、卖出十几到二十几笔）→ 小样本过拟合、bandit 难稳超基线的根源。
2. **1h ≠ 逐笔**：一根 1h 内 high/low 先后仍不可知（校准吻合 88–90%，残余失真已标注）。
3. **行情区间单一**：2024–2026 一段 regime，NVDA/TSLA 强相关、PDD 较独立；跨 regime 未必稳。
4. **human 回放非完全同口径**：真实成交只覆盖测试区间内 2025-01 起的一段；人类只在部分日子交易，策略每天可决策 —— 解读"bandit vs human"时需记得。

## 三、当前进度与已验证结论

| 阶段 | 内容 | 状态 | 关键数字 |
| --- | --- | --- | --- |
| **P0** | 数据底座（独立库 + 采集脚本 + H20 env） | ✅ | 见上表；H20 prefix env 已调通 |
| **P1** | 1h 撮合模拟器 + orders 校准 | ✅ | **校准吻合率 NVDA 90.3% / TSLA 87.8% / PDD 89.8%** |
| **P2** | 次日高低分位预测器 | ✅ | 分位**按股自适应收窄**（见下）：宽度降 ~25-30%，命中率守 ~50% |
| **P3** | 规则基线 + bandit 回测 | ✅ | 见下表 |
| **P3.1** | bandit 改进（超额奖励 + ε探索） | ✅ | NVDA 首次超 buy&hold；PDD 退化 |
| **P4** | 离线 RL（Discrete CQL，研究性） | ✅ 已跑 | **退化为"全程不动"** —— 见下，符合方案预判 |
| **P5** | 每日 HTML 报告 | ✅ | 自带**指标说明 + 每股分析总结**；`data/ml/reports/<date>/index.html` + latest |
| **P6** | 写入 web | ◻ 最后一步 | 达标稳定后才做 |

### P2 预测区间：按股自适应分位（收窄区间）
区间 `[L̂, Ĥ]` 由 low/high 两个分位张成：**分位越靠 0.5，区间越窄，但命中率越低**（宽度与覆盖率直接对赌，是分位回归的本质、非 bug）。原默认 0.10/0.90 覆盖 ~80% 极端波动，区间偏宽（TSLA 达 9.53%、阿里 10.49%），决策价值低。实测「宽度 vs 命中率」甜区后，按各股波动定档（`config.py::ALPHA_BY_CODE`）：

| 标的 | 分位(L/H) | 旧宽(0.10/0.90) | **新宽** | 命中率 |
| --- | --- | --- | --- | --- |
| US.NVDA | 0.20/0.80 | 6.63% | **5.48%** | 50% |
| US.TSLA | 0.20/0.80 | 9.53% | **6.77%** | 51% |
| US.PDD | 0.25/0.75 | 5.99% | **3.02%** | 56% |
| HK.00700 | 0.20/0.80 | 7.25% | **6.12%** | 54% |
| HK.09988 | 0.20/0.80 | 10.49% | **8.22%** | 57% |
| HK.01810 | 0.20/0.80 | 9.13% | **7.39%** | 48% |

**取舍**：多数股 0.20/0.80（宽度降 ~25-30%、命中率守 ~50%）；低波动的 PDD 激进到 0.25/0.75（命中率仍 56%、宽度砍到 3%）。命中率从 ~70% 降到 ~50% 是收窄的**诚实代价**——次日真实高/低约一半会戳出区间，命中率一并标在报告里，不藏波动风险。改档前须用 walk-forward 实测验证命中率不崩。

### P5 报告：自带「指标说明」+ 每股「分析总结」
- **指标说明**（顶部可折叠）：解释四条曲线（买入持有=地板 / 人类回放=现状坐标 / 规则 S0=门槛 / Bandit S2=被考核选手）如何对比着读，理想梯子 = Bandit ＞ 规则 ＞ 买入持有。
- **每股分析总结**（`report.py::_verdict`）：从四个策略的**实际期末净值规则化生成**一段文字（非 LLM、可复现），覆盖：Bandit 是否超买入持有 / 是否优于规则 / 人类回放定位 / 达成净值。照实写——多数股 Bandit 其实未跑赢买入持有（单边行情预期内），报告不替它粉饰。

### P3 → P3.1 回测对比（494 测试日，期末账户净值，初始 2 万，sklearn 后端）—— 诚实记录
| 标的 | rule(S0) | P3 bandit | **P3.1 bandit** | buy&hold | 行情 | P3.1 是否超 buy&hold |
| --- | --- | --- | --- | --- | --- | --- |
| NVDA | 25040 | 28858 | **34941** | 33638 | 单边强涨 | ✅ 超越（且超 rule） |
| TSLA | 28244 | 18606 | **25311** | 38750 | 单边强涨 | ✗ 改善但仍输 |
| PDD  | 19767 | 18974 | **12401** | 11840 | 震荡下行 | ✗ 退化（过度交易） |

**P3.1 改进**：奖励改为**相对 buy&hold 的超额**（直接对齐"打赢持有"）+ ε 探索 + reward 放大。

**诚实结论**：① 超额奖励在涨势显著改善——**NVDA 首次同时超 rule 和 buy&hold**；② 但 PDD（震荡下行）反而退化（超额奖励诱导过度交易、磨损）；③ **bandit 仍未在三支上稳定超基线** → go/no-go 仍**部分不达标**，明显是"按行情 regime 该用不同策略"的信号。按门槛：预测层(P2) + S0/超额-bandit 可交付，RL(P4) 与 regime-aware 策略作下一步研究方向，不强推单一数字。

### P4 离线 RL（Discrete CQL）结果 —— 诚实记录
在 H20 GPU 训练：行为策略（S0+ε=0.5）rollout 9440 transitions/标的 → Discrete CQL 训练 20k 步。

| 标的 | CQL 期末净值 | buy&hold | CQL 达成净值 | 学到的行为 |
| --- | --- | --- | --- | --- |
| NVDA | 20000（不变） | 33638 | 0 | **全程"不动"** |
| TSLA | 20000（不变） | 38750 | 0 | **全程"不动"** |
| PDD  | 20000（不变） | 11840 | 0 | **全程"不动"** |

**诚实结论**：CQL 在三支上均**退化为"全程不动"**（保守策略），未做任何交易。loss 分解印证：`conservative_loss(1.93) > td_loss(1.17)` —— CQL 的保守惩罚相对**弱奖励信号**过强，"不动"（零奖励、零风险）成了最优。**这与方案 §6/§9 对 RL 的预判完全一致**：数百条真实成交 / 万级 rollout 仍属小样本，离线 RL 在此规模上不产生有效交易策略。

**P4 定位（不变）**：研究性增量、**已如实验证不达标**。结论是科学的负结果——在当前数据量下，RL 不优于 S0/bandit/预测层，**不上线**。后续若要让 RL 有意义，需扩样本（更多标的/更长历史/更细粒度）或换更适配小样本的范式，非第一版范围。

### 实验总结（P0–P5 全程结论收口）

**一句话**：**预测层成立、撮合可信；决策层只在涨势局部有效；RL 在此数据量下无效。能交付的是"预测层 + 规则/超额-bandit + 每日报告"，RL 是诚实的负结果。**

| 模块 | 结论 | 证据 | 能否交付 |
| --- | --- | --- | --- |
| **预测层 P2** | ✅ **成立** | 次日区间命中 ~50%（自适应分位收窄后，宽度↓25-30%），walk-forward 无泄漏 | ✅ 可独立产出（复刻"预估高低点"） |
| **撮合 P1** | ✅ **可信** | 真实 orders 回放吻合 88–90% | ✅ 作为回测/奖励的可信底座 |
| **规则基线 S0** | ✅ **稳健对照** | 三支均给出合理净值，永久保留 | ✅ |
| **bandit P3.1** | ⚠️ **regime 依赖** | NVDA 超 buy&hold+rule；TSLA 改善仍输；PDD 退化 | △ 涨势可用、需 regime 感知 |
| **离线 RL P4** | ❌ **无效（负结果）** | CQL 三支全退化为"不动"，conservative_loss>td_loss | ✗ 不上线 |

**关键洞见**：
1. **拆两层的设计被验证是对的**——预测层（监督）稳、可独立交付；把不确定性关在决策层，避免了"端到端 RL 黑箱无法 debug"。
2. **目标函数口径很关键**——P3→P3.1 仅把奖励改成"相对 buy&hold 的超额"，NVDA 就从输变赢。说明**奖励对齐比换更复杂的模型更管用**。
3. **小样本是硬天花板**——同一套 1h 撮合 + 万级 rollout，RL（CQL）仍学不出有效策略；这不是调参问题，是数据量问题。**敬畏小样本的纪律被实证支持。**
4. **没有单一策略通吃**——涨势（NVDA/TSLA）buy&hold 极难超越；震荡/下行（PDD）择时才有价值。下一步真正值得做的是 **regime 感知**（按行情状态切策略），而非堆 RL。

**go/no-go 裁决**：按方案门槛，**停在 S1+S2 交付**（预测层 + 规则/超额-bandit + 每日报告），P4 RL 记为已验证的负结果。要继续提升，优先级 = regime 感知 ＞ 扩样本 ＞ RL。

## 五、运行：一个入口 `ml.sh` 搞定三件事

S0/P1/P2/P3.x/报告全是 CPU 算法，**本机 `mk` 环境即可，无需 GPU 机器**（P4 RL 才需 GPU）。

```bash
bash scripts/ml.sh data       # ① 例行更新数据（增量优先，见下）
bash scripts/ml.sh train      # ② 本地 Mac 训练/评估（P1 校准→P2 预测→P3 回测→生成报告）
bash scripts/ml.sh publish    # ③ 发布 HTML 报告到 www（公网）
bash scripts/ml.sh all        # 三步一条龙（默认；供 cron）

python -m pytest tests/test_ml_*.py -q   # 单测（48 passed）
```

> **增量缓存**：`data` 采集**增量优先**——库中已有近期数据（最新日期距今 ≤5 天）则只抓短窗（日线近 1 月 / 1h 近 5 天），UPSERT 与全量历史在库内合并，不重抓 5 年（例行从 ~30s 降到 ~11s）。
> 首次或需要补全历史：`python -m mystock.ml.fetch --full` 强制全量。库中全量历史不会被增量截断。

**例行（cron）** —— 工作日早 8 点（北京时间，约对应前一晚美股收盘后）：
```bash
crontab -e
0 8 * * 1-5  cd /Users/kk/Work/Workpace/GitHub/Seattle/myStock && bash scripts/ml.sh all >> data/ml/cron.log 2>&1
```
