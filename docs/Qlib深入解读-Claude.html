这份文档不是 README 的翻译。它是我逐层读过 qlib/data/ops.py、 qlib/data/dataset/、qlib/backtest/executor.py、 qlib/workflow/record_temp.py 之后，用自己的理解重写的一份「为什么这样设计」的解读—— 重点讲清楚 Qlib 每个抽象解决了量化研究里哪个具体的痛，以及它对我们这个 港股/美股本地持仓系统 myStock 意味着什么。

0 · 一句话看懂

Qlib 把「一个想法 → 一个可回测的量化策略」这条链路，拆成了一串可插拔、可复现、可缓存的标准化组件。 你换模型不用碰数据，换数据不用碰回测，换策略不用碰模型——每一段都通过一个统一的配置（YAML 或 Python 对象）拼起来。

本质 Qlib = 金融数据的表达式计算引擎 + ML 全流程的解耦框架 + 一个可复现的实验记录系统。 三者缺一，它就只是「又一个回测库」。

1 · Qlib 是什么

Qlib（发音 /kju:-lib/）是微软亚洲研究院开源的 面向 AI 的量化投资平台， Python 库名 pyqlib。它覆盖量化投资的完整链条： 因子挖掘（alpha）→ 风险建模 → 组合优化 → 订单执行， 并同时支持三种建模范式：监督学习（从行情里挖非线性规律）、 市场动态建模（应对概念漂移 / concept drift）、 强化学习（把连续交易决策建模成策略）。

它是它不是
研究到生产的 ML 量化流水线框架一个「买入信号提醒」App
金融时序数据的高性能存取 + 表达式计算引擎一个数据供应商（数据要自己灌）
一堆 SOTA 论文模型的统一复现基座（20+ 模型）一个开箱即用的实盘交易系统
可复现实验的记录/追踪系统（类 MLflow）持牌投顾 / 荐股工具

授权 MIT；主战场是 A 股（region: cn）与美股（region: us）， 但数据层是 region 无关的——港股/美股只要按格式灌进去就能用，这点对 myStock 很关键（见第 14 节）。

2 · 心智模型：一条流水线

理解 Qlib 最快的方式，是记住这条从原始数据到绩效报告的单向流水线。框架里几乎每个类都能对号入座：

原始行情 表达式引擎 数据管线 模型 信号→决策 执行→绩效 (csv/bin) ──► Expression ──► DataHandler ─► Dataset ─► Model ──► Strategy ──► Executor ──► Report 日线/分钟 $close/Ref/ 特征工程+标签 train/valid fit/predict Topk等 嵌套撮合 IC/年化/ 复权/停牌 Mean/Corr... +Processor /test 切分 → 预测分数 选股权重 限价滑点 回撤/夏普 (算子DAG) (Learn/Infer) (pred score) (order) (account) │ │ │ │ │ │ │ └── data ─────────┴── contrib.data ──┴── dataset ───┴── model ───┴─ strategy ─┴─ backtest ┴─ workflow.record

这条线的每一节都是「输入一个标准对象、输出一个标准对象」，所以任意一节都能被替换或单独跑。 Qlib 官方管这叫 loose-coupled modules——松耦合，每个组件都能 stand-alone。 下面我们把这条线拆开逐节看，重点是每一节到底在解决什么问题。

3 · 四层架构

官方把上面那条流水线在纵向上分成四层。这个分层的意义在于：越往下越通用、越稳定，越往上越贴近具体策略。

层职责关键组件
Infrastructure 基础设施 底层支撑：高性能数据存取、训练过程控制 DataServer（数据服务）、Trainer（训练器）
Learning Framework 学习框架 让「预测模型」和「交易智能体」可训练；统一监督学习与 RL 两种范式 Forecast Model、Trading Agent
Workflow 工作流 串起完整量化流程：抽数→预测→决策→执行 Information Extractor、Decision Generator、Execution Env
Interface 接口 面向用户的分析与报告 Graphical Reports、Analyser
读源码的抓手 分层是概念图，落到目录是：qlib/data/（基础设施）、qlib/model/+qlib/rl/（学习框架）、 qlib/workflow/+qlib/backtest/+qlib/strategy/（工作流）、 qlib/contrib/report/（接口）。qlib/contrib/ 是「社区/官方贡献的具体实现」， 模型、策略、数据 Handler 的现成货都在这里。

4 · 表达式引擎（灵魂）

如果只能记住 Qlib 的一个设计，就记这个。qlib/data/ops.py（1600+ 行）实现了一套 金融时序的表达式计算引擎：因子不是硬编码出来的，而是用字符串表达式声明出来的。 

# 一个「5日均线相对当前价的偏离」因子，直接写成字符串：
"Mean($close, 5) / $close - 1"

# 「过去20日 成交量与收益的相关性」：
"Corr($close/Ref($close,1)-1, Log($volume+1), 20)"

# 取数：给一批标的、一段时间、一列表达式，引擎自动求值
D.features(["SH600000"], ["$close", "Ref($close, 1)", "Mean($close, 5)"],
           start_time="2020-01-01", end_time="2020-12-31")

引擎里的算子分几大族，全是可组合的算子 DAG：

族代表算子语义
逐元素Abs Log Sign Power对单列逐点运算
二元/逻辑Add Sub Mul Div Gt Lt If And Or两列或列与常数运算、条件选择
时序 RollingRef Mean Sum Std Max Min Rank Quantile Skew Kurt沿时间轴滚动窗口聚合
时序回归类Delta Slope Rsquare Resi WMA EMA滚动线性回归斜率/残差、加权均线
成对 RollingCorr Cov两列滚动相关/协方差
横截面/重采样Rank(截面) TResample同一时刻跨标的排序、频率重采样
为什么这是灵魂
  • 因子即代码即数据：一个字符串就是一个可复现的因子，天然可版本管理、可批量生成。
  • 底层可加速：qlib/data/_libs/ 有 Cython 实现的 rolling/expanding，慢算子下沉到 C。
  • 可缓存：表达式求值结果按（表达式 × 标的 × 区间）缓存（qlib/data/cache.py）， 重复实验不重算。这就是 Qlib「跑第二遍飞快」的原因。
  • RD-Agent 的地基：因为因子是文本表达式，LLM 才能自动「写因子」（见第 12 节）。

两套现成因子库建在这个引擎上：Alpha158（158 个技术因子，价量衍生）和 Alpha360（360 维，近 60 日 OHLCV 原始序列展开）。它们就是 qlib/contrib/data/handler.py 里一组预定义表达式的集合。

5 · Handler / Dataset 数据管线

表达式引擎产出「特征矩阵」后，还有两个关键抽象把它变成「模型能直接吃的训练数据」。这两个抽象是 Qlib 里最容易搞混、但最值得理解的：

DataHandler：特征工程的容器

DataHandler 负责「从表达式 → 一张带特征列和标签列的大表」，并挂上一串 Processor 做预处理。 真正常用的是它的子类 DataHandlerLP（LP = Learn/Process），它的精髓是把预处理拆成两组：

处理器组作用典型算子
infer_processors推理期也要做的、无状态或用训练期参数的变换RobustZScoreNorm、Fillna、CSRankNorm
learn_processors只在训练期做的（可能会丢样本）DropnaLabel、样本过滤

于是同一份数据有两个「视图」：DK_I（infer，给预测用）和 DK_L（learn，给训练用）。 这解决了量化里一个隐蔽的 bug 源——训练时随手 dropna 掉的样本，推理时不能也丢（否则某天没预测）； 而归一化参数必须只用训练段拟合、推理段复用（否则未来信息泄漏）。Qlib 用 Learn/Infer 分离把这件事变成结构性保证。

常见 Processor

源码 qlib/data/dataset/processor.py 里现成的一批（CS 前缀 = Cross-Sectional 截面）：

  • ZScoreNorm / RobustZScoreNorm / MinMaxNorm——按列标准化（用训练段的均值方差/分位）。
  • CSZScoreNorm / CSRankNorm——截面标准化：同一天所有标的之间做 z-score 或排序归一，量化选股的常规操作。
  • DropnaProcessor / Fillna / ProcessInf——缺失与异常值处理。

Dataset：负责「切分」

DatasetH（H = 带 Handler）在 Handler 之上做时间切分： segments: {train, valid, test}。模型调用 dataset.prepare(["train","valid"], col_set=["feature","label"], data_key=DK_L) 就能拿到干净的 (x, y)。切分是 Dataset 的事、特征是 Handler 的事，两者正交—— 所以你可以固定一份 Alpha158 Handler，滚动地换 train/test 窗口（这正是第 11 节滚动训练的基础）。

关键点 Qlib 数据是 (datetime, instrument) 双层 MultiIndex 的 DataFrame。列分组是 feature / label / weight。理解这个索引结构， 读任何 Qlib 数据代码都会顺很多。

6 · 模型抽象与模型园

模型层的抽象极简，这是它能容纳 20+ 种模型的原因。qlib/model/base.py：

class Model(BaseModel):
    def fit(self, dataset: Dataset, reweighter=None):   # 训练
    def predict(self, dataset: Dataset) -> pd.Series:   # 输出每个(日期,标的)的预测分数

只要实现这两个方法，任何模型都能接进 Qlib 流水线——它不关心你内部是 GBDT 还是 Transformer， 只要 predict 吐出「分数」。这个「分数（score / signal）」就是连接模型层与策略层的唯一契约。

内置模型园（qlib/contrib/model/ + examples/benchmarks/）

类别模型
树模型LightGBM、XGBoost、CatBoost、DoubleEnsemble
线性Linear
经典时序 NNMLP、LSTM、GRU、ALSTM、TCN、SFM
Transformer 系Transformer、Localformer、TFT
图/关系GATs、HIST、IGMTF（引入股票间关系/概念图）
自适应/前沿ADARNN、ADD、TRA、TCTS、TabNet、KRNN、Sandwich
经验之谈 Qlib 官方 benchmark 里，在 Alpha158/Alpha360 上，调好的 LightGBM 往往打得过大多数深度模型， 且训练是分钟级。深度模型的价值更多在于能吃 Alpha360 那种原始序列、能建关系结构。 对 myStock 这种个人系统，树模型是性价比最高的起点。

7 · Workflow 与记录链

Qlib 的实验以 Experiment → Recorder 组织（qlib/workflow/，接口类似 MLflow）。 R 是全局记录器句柄，一次 with R.start(...) 就是一次可复现的实验， 模型、参数、预测、绩效全部落盘、可回捞。

真正串起「预测 → 分析 → 组合回测」的是 record_temp.py 里的一条记录链，它们按顺序执行、后者依赖前者的产物：

Record做什么产物
SignalRecord用训练好的 model 在 test 段跑 predictpred.pkl（预测信号）
SigAnaRecord对信号本身做分析（不涉及交易）IC、Rank IC、ICIR、多空收益
PortAnaRecord把信号喂给策略 + 回测，做组合分析年化、夏普、最大回撤、超额收益
这套设计的价值 信号质量（IC）与策略绩效（夏普）被拆成两级评估。一个模型 IC 高但回测差， 通常是策略/成本/换手的问题，不是模型的问题——这条链让你能分别定位。
IC（信息系数）= 预测值与真实收益的截面相关；ICIR = IC 的均值/标准差，衡量稳定性。

8 · 回测：嵌套执行器

这是 Qlib 第二个真正有设计感的地方（qlib/backtest/executor.py）。它把回测抽象成 「策略 Strategy 出决策、执行器 Executor 撮合」的循环，而且执行器可以嵌套。

日频层： Strategy(日) ──决策(目标持仓)──► NestedExecutor(日) │ 把「今天要调仓」拆成日内订单 ▼ 分钟层： Strategy(日内) ──订单──► SimulatorExecutor(分钟) │ 逐分钟撮合，算滑点/手续费/涨跌停 ▼ 账户 Account（现金/持仓/成交）

NestedExecutor 里嵌一个更细粒度的「策略+执行器」子循环， SimulatorExecutor 是最底层的真实撮合。这套结构一次性解决了两个平时很别扭的问题：

  • 多频率协同：日频选股策略 + 日内 VWAP/TWAP 执行，可以在同一个回测里嵌套跑，收益归因清清楚楚。
  • 决策与执行解耦：策略只管「我想持有什么」（TradeDecision）， 执行器负责「在这个市场约束下怎么成交」——RL 订单执行（第 10 节）就是把最底层这个执行器换成一个学出来的智能体。

交易约束在 Exchange 里配：limit_threshold（涨跌停不可成交）、 deal_price（用 close/vwap）、open_cost/close_cost/min_cost（买卖佣金、最低费）。 这些直接决定回测可信度。

9 · 策略：TopkDropout 解剖

qlib/contrib/strategy/signal_strategy.py 里最经典的是 TopkDropoutStrategy， 几乎所有 benchmark 都用它。它把「预测分数」变成「实际持仓」，机制值得逐字理解：

参数含义
topk目标持仓只数（如 50 只）
n_drop每期最多换掉几只（如 5 只）
signal打分来源（<PRED> = 上游模型预测）

每个调仓日的逻辑：

  1. 按预测分数对全市场排序。
  2. 看当前持仓里，分数掉出前列的，最多卖掉 n_drop 只（不是全部重排）。
  3. 用排名最靠前、当前没持有的标的补上，补齐到 topk 只。
  4. 等权（或按 WeightStrategyBase / EnhancedIndexingStrategy 优化权重）持有。
为什么是 dropout 而不是全换 每期只换 n_drop 只，是为了压低换手率——换手直接吃交易成本， 也让回测的年化不至于被「理想化的天天满仓重排」虚高。这是一个「工程上很聪明」的默认策略： 简单、可解释、换手可控。它也解释了为什么调 n_drop 对回测结果影响巨大。

另有 EnhancedIndexingStrategy（增强指数，控制相对基准的行业/风格暴露）和 strategy/optimizer/（组合优化器，做风险约束下的权重求解）。

10 · 强化学习与订单执行

qlib/rl/ 是 Qlib 的第三种范式。它最成熟的落地场景是订单执行（order execution）： 给你一个「今天要买 100 万股 A」的目标，怎么在一天内切成小单、择时下单，让平均成交价尽量优于 VWAP。

RL 组件与标准 Gym 范式对齐，但为金融做了拆分：

组件Qlib 类金融含义
环境模拟Simulator市场撮合（复用第 8 节的执行器）
状态解释StateInterpreter把市场状态编码成观测向量
动作解释ActionInterpreter把策略动作翻译成订单
奖励Reward成交价 vs 基准的差（越省越好）
策略Policy（PPO 等）可训练的交易智能体
插一句 这套 Simulator / StateInterpreter / ActionInterpreter / Reward / Policy 的拆法， 和我们 myStock 里 mystock/ml/ 正在做的 policy.py、reward.py、 regime.py、calibrator.py 思路是同源的——都是把「决策」拆成可独立替换、可单测的纯组件。

11 · 市场漂移与在线滚动

金融数据最难的地方：分布会漂移（concept drift）。去年有效的因子今年可能反向。 Qlib 给了两套工具应对：

① 滚动训练（Rolling）

qlib/contrib/rolling/：不训练「一个」模型，而是随时间滚动地不断重训—— 每隔一段用最近数据重新 fit，模拟真实世界里「模型要定期再训」的节奏。评估更诚实。

② 元学习 / DDG-DA

更进一步，把「未来数据分布会怎么变」也建模出来（meta-learning，qlib/contrib/meta/）， 在重训时主动补偿漂移，而不是被动追随。代表工作 DDG-DA。

③ 在线服务（Online Serving）

qlib/workflow/online/：管理「当前上线的是哪个模型」，自动滚动更新、灰度切换—— 把研究期的滚动训练，延伸成生产期的模型生命周期管理。

对个人系统的提醒 漂移是量化里比「模型不够强」更致命的问题。任何在历史上惊艳的回测， 不做滚动/样本外验证都要打对折看。myStock 若引入 ML，滚动回测应是默认口径，而非可选项。

12 · RD-Agent：自动因子工厂

Qlib README 现在头条推的是 RD-Agent—— 一个用 LLM 驱动的自动因子挖掘 + 模型优化智能体框架。它能做两件事：

  • 从想法/研报里读因子：把券商研报里描述的因子逻辑，自动翻译成 Qlib 表达式并回测。
  • 自动迭代：提出因子 → 回测 → 看结果 → 改进 → 再回测，形成 R&D 闭环（论文 R&D-Agent-Quant, arXiv:2505.15155）。
为什么第 4 节的表达式引擎是前提 正因为 Qlib 的因子是文本表达式、且回测是标准化可自动跑的，LLM 才能把「写因子」变成一个可自动闭环的任务。 这是 Qlib 早年那个表达式设计，在 LLM 时代兑现的复利。

13 · 一份 YAML 全景

前面讲的所有组件，最终由一份 YAML 声明式地拼起来，用 qrun config.yaml 一键跑完 「初始化 → 建数据集 → 训练 → 预测 → 信号分析 → 组合回测 → 落盘」。 下面是官方 LightGBM×Alpha158 配置，逐块对应到前面的章节： 

qlib_init: {provider_uri: "~/.qlib/qlib_data/cn_data", region: cn}
market: &market csi300              # 股票池
benchmark: &benchmark SH000300      # 对标基准

data_handler_config: &dhc          # ← 第 5 节：Handler
    start_time: 2008-01-01
    fit_start_time: 2008-01-01      # 归一化参数只用这段拟合（防泄漏）
    instruments: *market

port_analysis_config: &pac          # ← 第 8/9 节：策略 + 回测
    strategy:
        class: TopkDropoutStrategy
        kwargs: {signal: <PRED>, topk: 50, n_drop: 5}
    backtest:
        account: 100000000
        exchange_kwargs:            # 交易成本/涨跌停
            limit_threshold: 0.095
            deal_price: close
            open_cost: 0.0005
            close_cost: 0.0015

task:
    model:                          # ← 第 6 节：模型
        class: LGBModel
        module_path: qlib.contrib.model.gbdt
        kwargs: {learning_rate: 0.2, num_leaves: 210, max_depth: 8, ...}
    dataset:                        # ← 第 5 节：Dataset 切分
        class: DatasetH
        kwargs:
            handler: {class: Alpha158, kwargs: *dhc}
            segments:               # 训练/验证/测试三段
                train: [2008-01-01, 2014-12-31]
                valid: [2015-01-01, 2016-12-31]
                test:  [2017-01-01, 2020-08-01]
    record:                         # ← 第 7 节：记录链
        - {class: SignalRecord}     # 生成预测
        - {class: SigAnaRecord}     # IC 分析
        - {class: PortAnaRecord, kwargs: {config: *pac}}  # 组合回测

一份 YAML 就是一次完整实验。想换模型？改 model.class。想换股票池？改 market。 想换回测成本？改 exchange_kwargs。组件解耦的价值，在这份配置里最直观。

纯 Python 写法见 examples/workflow_by_code.py，逻辑等价，适合要在流程里插自定义步骤时用。

14 · 对 myStock 的启示

Qlib 是「工业级研究框架」，myStock 是「个人本地持仓系统」，定位不同，但有几处理念可以直接借鉴：

值得借鉴
  • 表达式化因子：把技术指标写成可组合、可缓存的表达式，而非散落的函数——利于批量试因子。
  • Learn/Infer 分离：归一化只用训练段拟合，是防未来泄漏的结构性保证。
  • 信号↔策略两级评估：IC 看信号、夏普看策略，分别定位问题。
  • 滚动回测为默认：对抗漂移，别信单段回测。
  • 决策组件化：正对应我们 ml/policy.py、reward.py、regime.py 的拆法。
不必照搬
  • 整套引入过重：Qlib 假设有完整历史行情数据服务；myStock 只有富途+yfinance 的本地 SQLite 快照。
  • 数据边界冲突：Qlib 要 region 全市场 bin 数据，我们是「已持仓标的」的窄数据。
  • Web 只读边界：myStock 的 CLAUDE.md 明确 Web 只读 SQLite；Qlib 的重计算应放在 pipeline 侧离线跑，绝不进 Web 请求路径。
  • 港股/美股：Qlib 主战场是 A 股；港美股要自己灌数据、自建 instruments 列表。
务实结论 与其整体接入 Qlib，不如借其设计、造轻量版：在 mystock/ml/ 里沿用「表达式因子 + Learn/Infer 分离 + 滚动回测 + 信号/策略两级评估」的方法论， 用我们自己的 SQLite 数据跑。若某天需要认真做多模型 benchmark，再考虑把导出的数据喂给独立的 Qlib 环境离线研究—— 与 myStock 的 Web 只读 边界不冲突。

15 · 坑与边界

坑说明
数据要自己准备Qlib 只给下载脚本（scripts/get_data.py 拉公开样例）；自有市场/私有数据要转成 Qlib bin 格式。
未来信息泄漏归一化拟合区间、标签的时间对齐（Ref($close, -N) 用未来收益做标签）极易出错。切分与 fit 区间必须严格。
回测≠实盘涨跌停不可成交、滑点、最低费、停牌、复权——exchange_kwargs 配错，回测会系统性偏乐观。
缓存陷阱表达式缓存加速的同时，改了底层数据但没清缓存会读到旧结果。调试时留意 ~/.qlib 缓存。
深度模型收益边际多数场景 LightGBM 已够好；上深度模型前先确认数据/因子瓶颈不在别处。
不是投顾Qlib 是研究工具，输出的是「研究信号」，不构成投资建议——与 myStock「不输出个性化投资建议」的定位一致。
